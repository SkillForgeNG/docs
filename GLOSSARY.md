# SkillForge Glossary

Common terms, acronyms, and definitions used throughout the SkillForge project.

## Learning & Education Terms

### Competency
A measurable ability or skill that learners develop through the platform. Competencies are tracked and assessed through projects and assessments.

### Learning Path
A curated sequence of courses, resources, and projects designed to help learners develop specific competencies. Learning paths are project-based and skills-focused.

### Learning Journey
The complete progression of a learner through one or more learning paths, including all completed and in-progress modules.

### Project-Based Learning (PBL)
A learning methodology where learners develop competencies through working on real-world projects, applying knowledge in practical contexts.

### Curated Content
Educational resources (courses, tutorials, articles, videos) selected and organized by SkillForge content experts to support specific learning paths.

### Assessment
Evaluation methods used to measure learner progress and competency achievement, including quizzes, projects, and peer reviews.

### Rubric
A detailed set of criteria used to evaluate and score learner work and projects.

### Microlearning
Short-form learning modules designed to be completed in 5-15 minutes, focusing on specific skills or concepts.

## Platform Terms

### User
Any person using the SkillForge platform, including learners, instructors, and administrators.

### Learner
A user actively engaged in learning through courses, projects, and competency development.

### Instructor
A user responsible for creating, curating, or managing learning content and providing guidance.

### Learning Management System (LMS)
The platform infrastructure that manages course delivery, learner progress, and assessments.

### Content Library
The collection of all curated educational resources available on the SkillForge platform.

### Dashboard
The personalized view showing learner progress, recommended paths, and available courses.

## Technical Terms

### API
Application Programming Interface - allows systems to communicate and exchange data.

### Backend
Server-side application that handles business logic, data processing, and database management.

### Frontend
Client-side application that users interact with directly, built with web technologies.

### Database
Structured storage system for all platform data including users, courses, and progress.

### Authentication
Process of verifying user identity, typically through username/password, OAuth, or SSO.

### Authorization
Process of determining what authenticated users are allowed to access or do.

### Microservice
Small, independent service that performs a specific function and communicates with other services via APIs.

### Scalability
Ability of the system to handle increased load by adding resources or distributing work.

## Project Management Terms

### Sprint
Fixed time-boxed iteration (typically 2 weeks) during which the team commits to completing specific work.

### Sprint Planning
Meeting at the start of each sprint to select work items and plan implementation.

### Sprint Retrospective
Meeting at the end of each sprint to reflect on process improvements.

### User Story
Description of a feature from the user's perspective, typically formatted as "As a [user type], I want [capability] so that [benefit]".

### Story Points
Estimate of work complexity and effort required to implement a feature.

### Backlog
Prioritized list of features, improvements, and fixes not yet scheduled for a sprint.

### Roadmap
High-level plan showing major features and releases over time.

## Organizational Terms

### Stakeholder
Person or group with interest in the project's success, including users, sponsors, and team members.

### Pull Request (PR)
Request to merge code changes into the main codebase, subject to review and approval.

### Code Review
Process where team members examine code changes for quality, correctness, and adherence to standards.

### Architecture Decision Record (ADR)
Document recording a significant technical decision, including context, alternatives, and rationale.

### Documentation
Written explanations, guides, and specifications for code, features, and processes.

### Open Source
Software development model where code is publicly available and community contributions are encouraged.
